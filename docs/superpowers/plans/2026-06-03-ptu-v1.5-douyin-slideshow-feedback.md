## 用户原始反馈

用户测试了当前“仿抖音式视频”生成能力，并反馈：

1. 第一张和第二张素材切换时，没有抖音图文/实况那种过渡动画。

3. 抖音的行为是：图片或实况素材按固定节奏播放，素材播放完后如果背景音乐还没结束，会回到第一张继续循环，直到音乐结束。

## 明天优先修复事项

优先级 1：隐藏所有生成视频相关命令符窗口

- 点击“生成视频”时，不应该弹出 cmd / PowerShell / FFmpeg 控制台窗口。
- 子进程应该后台运行。
- Windows 打包态需要使用 `subprocess` 的 `creationflags=CREATE_NO_WINDOW` 或等价封装。
- 需要覆盖：
  - 主 slideshow 合成 FFmpeg 调用。
  - 实况预处理 / live_0000 合成 FFmpeg 调用。
  - ffprobe 或其它探测调用，如果会弹窗也要隐藏。
- 验收：用户点击生成视频时前台不出现命令符窗口；生成结束后也没有残留窗口。

优先级 2：修复 FFmpeg 子进程输出解码

- 当前日志出现 GBK 解码错误。
- 建议统一对子进程输出使用 `encoding="utf-8"` + `errors="replace"`，或直接按 bytes 捕获后安全解码。
- 目的：不让 FFmpeg 的中文/特殊字符输出导致 `_readerthread` 报错。
- 验收：生成视频时日志不再出现 `UnicodeDecodeError`。

优先级 3：抖音式固定预设，而不是可配置编辑器

- 用户明确不要窗口选择、淡入淡出选择、转场选项。
- UI 应该保留一个简单入口：一键生成抖音式视频。
- 内部固定 preset：
  - 自动判断素材数量。
  - 自动按抖音图文/实况节奏切换。
  - 素材结束但音乐未结束时，从第一张重新循环。
  - 最终输出时长以背景音乐为准。
- 如果没有背景音乐，再采用合理默认时长，但这不是当前主路径。

优先级 4：补齐抖音式过渡动画和节奏

- 当前参考屏录不是简单 fade，也不是让用户选转场。
- 目标更像抖音图文/实况播放中的固定滑动/滚动切换体验。
- 已初步量化：当前 Ptu 约 6 秒一换，参考过渡主体约 0.3-0.5 秒，Ptu 缺少横向推入/推出动画。
- 明天仍需继续复核：
  - 固定单页停留是用 3 秒、3.5 秒还是其它值。
  - 切换方向是否始终从右往左。
  - 缓动是否需要 ease-out，而不是线性匀速。
  - 实况素材内部播放时长与静态图停留时长的区别。
- 初步方向：
  - 先做一个固定 `douyin` preset。
  - 使用 30 fps。
  - 固定转场时长先按约 0.35 秒实现，再按视觉效果微调。
  - 固定每页播放节奏先按约 3 秒静态页停留实现，再按参考视频和用户观感微调。
  - 循环序列直到音乐结束，并裁剪到音乐结束点。

优先级 5：重新验证实况合成

当前代码判断：

- 前端二维码显示后，每 2 秒调用 `/api/login/confirm` 轮询。
- 后端 `confirm_login()` 在 Playwright 模式下主要通过 `_confirm_pw()` 检查 headless 页面上下文里的 cookie。
- Playwright 获取二维码时，当前代码只保存了二维码图片，没有保存二维码接口返回的真实 token。
- 因此 Playwright fallback 模式无法稳定走“token 轮询二维码状态”的路径。
- 如果抖音扫码后没有把登录态写入这个 headless 页面 cookie，后端就会一直返回 `waiting`。
- 前端对 `waiting` 没有明显状态更新，对异常也基本吞掉，所以用户看到的就是“扫码后没任何反应”。

初步结论：

- 这是一个登录确认链路问题，不是二维码展示问题。
- v1.4.2 已复现该风险；当前 v1.5 代码仍需要复核并修复同类逻辑。
- 明天修复时应把它和“命令符窗口隐藏”一起列为高优先级用户测试阻塞。

明天建议修复方向：

1. Playwright 捕获 `/passport/web/get_qrcode` 响应时，除了 `qrcode`，还要保存 `token` / `verify_ticket` / `passport_ticket` 等二维码状态字段。
2. `/api/login/confirm` 应优先使用真实 token 调用 SSO 的二维码状态检查接口，而不是只检查页面 cookie。
3. 如果状态是“已扫码，待手机确认”，前端文案应显示“已扫码，请在手机上确认”。
4. 如果状态是“二维码过期”，前端应显示“二维码已过期，请刷新”。
5. 如果一直 waiting，日志应周期性记录关键状态，便于远程排障。
6. 前端不要吞掉登录确认错误，至少写入日志或显示短错误。
7. 修复后用朋友这类干净机器路径复测：二维码显示 -> 手机扫码 -> 手机确认 -> Ptu 登录态变为已登录 -> cookies 写入 `%LOCALAPPDATA%\Ptu\cookies.yaml`。

### 登录体验优化

目标：让小白用户知道登录流程卡在哪一步，而不是看到二维码后“扫码了也没反应”。

建议状态分层：

- `请扫码`：二维码已生成，等待用户用抖音扫码。
- `已扫码，请在手机上确认登录`：后端已经检测到二维码被扫码，但手机端尚未确认。
- `登录成功`：手机端确认成功，Ptu 已拿到并保存登录 cookie。
- `二维码已过期，请刷新`：二维码 token 过期，不能继续等待。
- `网络异常，请稍后重试`：抖音接口、网络或登录轮询失败。

建议 UI 行为：

- 二维码下方显示倒计时，例如 `二维码将在 120 秒后过期`。
- 二维码过期后，按钮改为 `刷新二维码`。
- 可以自动刷新一次二维码，但仍保留手动刷新按钮。
- 手机扫码后立即把文案从 `请用抖音扫描二维码` 改成 `已扫码，请在手机上确认登录`。
- 手机确认成功后显示成功图标和 `登录成功`，再自动关闭登录弹窗。
- 如果用户长时间停在“已扫码未确认”，提示 `请检查手机抖音是否弹出确认登录`。
- 前端不要吞掉 `/api/login/confirm` 的异常；至少显示短提示，并写入日志。

建议日志行为：

- 记录二维码生成成功。
- 记录开始轮询扫码状态。
- 记录已扫码但未确认。
- 记录二维码过期。
- 记录确认成功和 cookies 写入路径。
- 记录异常原因，但不要记录真实 cookie 内容。

- 右侧只作为可见抖音浏览器使用。
- 用户自己在右侧正常浏览、搜索、点用户、点作品。
- 用户自己复制当前作品或主页链接。
- 左侧继续使用现有 `粘贴` / `抓取` / `主页抓取` 能力。
- Ptu 第一版不自动读取页面、不自动扫描当前页、不自动抓右侧内容。
- 这样能满足“不用手机端来回复制”的主要需求，同时明显降低自动化和风控风险。
- 等这一版稳定后，再考虑加 `抓取当前页` 这类半自动按钮。

账号风险边界：

- 用户明确担心账号被抖音封禁。
- 该功能必须按低风险原则设计。
- Ptu 不做后台自动刷推荐流。
- Ptu 不做自动点赞、评论、关注、私信、发布、转发。
- Ptu 不做高频批量访问。
- Ptu 不绕过验证码、风控、人机校验。
- Ptu 不保存抖音账号密码。
- Ptu 不把 cookie 上传到任何服务器。
- Ptu 只在用户主动点击时读取当前页面并抓取。
- 批量下载必须有明确数量限制、节流和中止按钮。
- 失败或风控提示出现时，应停下来提示用户，不继续重试冲接口。

建议实现原则：

1. `用户主动操作优先`：右侧浏览由用户自己点，Ptu 只响应用户点击的“抓取当前页”。
2. `只读当前上下文`：读取当前作品链接、主页链接或页面 URL，不自动扫全站。
3. `低频节流`：批量下载主页作品时控制并发和间隔。
4. `可停止`：所有批量任务都能取消。
5. `不碰账号行为`：不做互动类自动化。
6. `可解释`：界面提示“请像正常浏览一样操作，Ptu 只抓取你当前打开的作品或主页”。

阶段建议：

- 不放进明天第一批小修。
- 可作为 v1.6 或 v1.5 后半段的大功能。
- 第一版优先做 `右侧可见浏览 + 用户手动复制链接 + 左侧粘贴抓取`。
- 后续再做 `抓取当前页`，并在独立设计文档中定义浏览器面板、URL 读取、登录态复用、风险边界和测试方式。
