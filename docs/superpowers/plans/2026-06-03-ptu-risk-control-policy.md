# Ptu 风险管控红线与低风险产品策略

记录时间：2026-06-03

适用范围：Ptu 的扫码登录、内置浏览、单作品抓取、主页抓取、批量下载、日志导出、后续“抓取当前页”等能力。

强制要求：任何新会话、任何新 Agent、任何后续开发计划，开始改 Ptu 前必须先读本文件。若需求与本文件冲突，以本文件为准，先向用户说明风险，再调整方案。

## 依据与背景

本文件不是法律意见，而是 Ptu 的产品与工程安全边界。它基于：

- 抖音开放平台开发者服务协议：
  `https://open.douyin.com/platform/resource/docs/operation-standard/agreement-protocol`
- 抖音开放平台移动应用 / 网站应用服务协议：
  `https://partner.open-douyin.com/docs/resource/zh-CN/dop/operation-standard/service-protocol/mobile-web-app-protocal`
- DouyinCrawler 参考项目：
  `https://github.com/erma0/douyin`
- 本地参考包：
  `C:\Users\ASUS\Desktop\AI\GitHub下载\DouyinCrawler_v5.1.260425\DouyinCrawler_v5.1.260425`

从协议和参考项目得到的核心判断：

- 平台协议强调应按规则和授权使用接口、功能和数据，不应未经许可获取、存储、披露用户信息。
- 协议对非正常浏览、蜘蛛、爬虫、拟人程序等方式获取平台信息有明确风险提示。
- 协议要求用户数据处理应取得授权、只获取必要数据，并提供删除路径。
- DouyinCrawler 是全量采集型工具，覆盖主页、喜欢、收藏、话题、音乐、搜索、关注、粉丝、增量采集和批量下载；其 GitHub 仓库已在 2026 年 6 月因合规审查清空源码。
- Ptu 不能走“全量采集器”路线，必须定位为用户主动操作下的本地桌面辅助工具。

## 产品定位红线

Ptu 的低风险定位：

- 用户本机运行。
- 用户主动登录。
- 用户主动复制链接。
- 用户主动点击抓取、下载、生成视频。
- Ptu 只处理用户明确提供或当前主动打开的作品 / 主页。
- Ptu 不上传 cookie、不上传账号信息、不把用户数据转移到第三方服务器。

Ptu 不能被设计成：

- 后台自动刷抖音。
- 大范围采集器。
- 账号行为自动化工具。
- 搜索 / 关注 / 粉丝 / 喜欢 / 收藏 / 话题 / 音乐原声全量采集工具。
- 绕过平台风控、验证码、人机校验的工具。

## 绝对不要做

以下功能禁止实现。如果用户提出类似需求，先解释风险，并改为低风险替代方案。

1. 不做自动点赞、评论、关注、取消关注、私信、转发、发布、收藏等账号互动行为。
2. 不做后台自动刷推荐流、自动翻页、自动扫用户主页、自动扫搜索结果。
3. 不做关注列表、粉丝列表、喜欢列表、收藏列表的采集入口。
4. 不做话题挑战、关键词搜索、音乐原声、合集的全量采集入口。
5. 不做大规模增量采集，不维护“长期监控某用户新作品自动下载”的能力。
6. 不绕过验证码、人机验证、风控页、登录限制、地区限制或接口限制。
7. 不连续高频重试被 403、风控、验证拦截的接口。
8. 不使用代理池、账号池、cookie 池、批量账号轮换。
9. 不保存抖音账号密码。
10. 不把 cookie 写进可上传、可发布、可导出的文件。
11. 不把 cookie 明文放进 Git、Release、安装包、日志、诊断包或问题报告。
12. 不宣称 Ptu 是抖音官方工具、合作工具或获得平台特别授权。
13. 不复制 DouyinCrawler 的全量采集能力作为 Ptu 功能方向。

## 必须包括

任何涉及抖音登录、浏览、抓取、下载的新功能，必须同时包含以下控制。

### 用户主动触发

- 抓取必须由用户点击按钮触发。
- 下载必须由用户点击按钮触发。
- 生成视频必须由用户点击按钮触发。
- 右侧内置浏览第一版只作为可见浏览器，不自动读取页面、不自动扫页面。
- 后续如做 `抓取当前页`，只能读取用户当前正在看的作品 / 主页 URL。

### 数量和频率限制

- 单次主页批量下载必须有最大数量限制。
- 批量下载必须有并发上限。
- 批量下载必须有请求 / 下载间隔。
- 下载失败必须有重试上限。
- 出现连续失败必须停止，不继续压接口。
- 出现 403、验证码、风控页、登录异常时必须停止并提示用户。

建议初始默认值：

- 单次主页最多处理 30 个作品，后续如要提高，必须给用户明确确认。
- 下载并发不超过 3。
- 同一主页批量任务之间至少间隔 1-2 秒。
- 单个下载最多重试 2-3 次。
- 连续 3 次风控类失败后停止整批任务。

### 可取消和可恢复

- 批量下载必须有取消按钮。
- 生成视频必须能取消或至少能清楚提示正在处理。
- 失败后应保留已完成结果，不强制重新开始所有任务。

### 登录与 cookie 安全

- cookie 只保存在本机运行数据目录。
- cookie 文件不得进入 Git 跟踪。
- cookie 不得进入安装包。
- cookie 不得进入导出的日志 / 诊断包。
- 必须提供退出登录 / 清除登录状态能力。
- 登录状态要显示清楚：未扫码、已扫码待确认、登录成功、二维码过期、异常。
- 不保存账号密码，不代理用户输入账号密码。

### 日志与诊断

- 日志可以记录路径、版本、状态、错误类型。
- 日志不得记录真实 cookie、账号密码、完整敏感请求头。
- 一键导出诊断包必须过滤敏感信息。
- 诊断包建议包括：
  - Ptu 版本。
  - 安装路径。
  - 是否打包态。
  - Chromium / FFmpeg 是否存在。
  - cookies 文件是否存在。
  - 当前登录状态。
  - 最近日志。

### 用户提示

- 内置浏览区应提示：请像正常浏览一样操作，Ptu 只处理你主动复制或主动抓取的当前页面。
- 批量下载前应提示数量。
- 风控或验证码出现时应提示用户停止操作，不继续自动重试。
- 不用“免责声明”替代实际限流、取消、停止和日志过滤。

## 内置抖音浏览面板策略

推荐第一版：

- 右侧为可折叠抖音浏览面板。
- 用户在右侧正常浏览抖音。
- 用户自己复制作品或主页链接。
- 左侧继续使用 `粘贴`、`抓取`、`主页抓取`。
- Ptu 不自动扫描右侧页面，不自动读取作品列表，不自动批量抓当前页面。

后续可选增强：

- `复制当前链接到左侧`：读取当前浏览器 URL，填入左侧输入框。
- `抓取当前作品`：只对当前作品 URL 生效。
- `抓取当前主页`：只对当前主页 URL 生效，并保留数量上限。

后续增强必须满足：

- 用户点击按钮触发。
- 只读取当前 URL。
- 不自动滚动页面。
- 不自动翻页。
- 不自动从 DOM 中扫全量作品。
- 遇到风控/验证码立即停止。

## DouyinCrawler 参考结论

可借鉴：

- 下载队列。
- 并发上限。
- 重试上限。
- 下载超时。
- WebView 持久化会话。
- GUI / 后端 / 下载器分层。

不照搬：

- 关注 / 粉丝采集。
- 喜欢 / 收藏采集。
- 话题 / 搜索 / 音乐原声采集。
- 增量监控采集。
- 全量 REST API 暴露给用户。
- 明文 cookie 配置。
- 用免责声明替代产品控制。

## 开发验收门槛

涉及抖音登录、浏览、抓取、下载的新改动，验收前必须检查：

- 是否有用户主动触发按钮。
- 是否有数量上限。
- 是否有并发上限。
- 是否有重试上限。
- 是否遇到风控 / 验证 / 403 即停止。
- 是否提供取消按钮。
- 是否过滤 cookie 和敏感请求头。
- 是否没有新增全量采集入口。
- 是否没有新增账号互动自动化。
- 是否没有把 cookie 纳入 Git / 安装包 / Release / 日志导出。

建议新增或更新自动检查：

- `scripts/release_check.py` 检查敏感文件和 cookie 导出风险。
- 测试覆盖诊断导出不包含 cookie。
- 测试覆盖批量任务默认数量和并发上限。
- 测试覆盖风控类错误会停止重试。
